# Alembic metadata drift classification

Measured: 22 July 2026

Database: empty PostgreSQL 16 database upgraded through the single
`0063_clinical_document_institution_provenance` head.

## Result

`alembic check` reports the historical ORM/migration comparison operations documented below. The migrated PostgreSQL
schema is internally consistent and is the deployment source of truth. The
output is historical ORM/migration metadata drift, not evidence that an empty
upgrade or restore omits an application table. No broad comparison ignore and
no autogenerated migration is introduced.

The findings fall into the groups below. The object lists are exhaustive for
the current `alembic check` output.

The previous count of 86 was measured at revision 0062. Revision 0063 adds the
nullable `clinical_documents.institution_id` foreign key and its explicit
index; its ORM metadata matches the migration. A final post-remediation count is
recorded only after the closing validation run, rather than inferred here.

## Backup and Module 4 impact

- Empty upgrade to head succeeds.
- Downgrade from 0063 to 0062 and re-upgrade to 0063 succeeds; the broader
  downgrade to 0060 and re-upgrade through the head remains supported.
- The drift does not omit data from `pg_dump`/restore because PostgreSQL, not
  ORM metadata, owns backup contents.
- There is no verified drift blocker for backup/recovery work.
- A future metadata-alignment track may declare migration-owned indexes and
  constraints in ORM metadata, but it must compare each exact name and must not
  emit the destructive autogenerated operation list.

The long-lived local development database remains at revision 0061. Its value
is not assumed and it was not modified. It is not a canonical validation
database; upgrade requires a separately verified local backup.
